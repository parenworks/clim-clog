#+TITLE: clim-clog - Design & Roadmap
#+OPTIONS: toc:nil num:nil

This document records /what we are building, why, and in what order/, so the
investigation stays legible as it grows.

* The question we are answering

Fade's framing: a CLIM-based "Lisp machine" UI implies CLIM, but a web
interface "would require a side project to map CLIM onto websockets + CLOG".
This project is that spike. The concrete question:

#+begin_quote
Can a CLIM medium drive a browser canvas through CLOG cleanly enough to be
worth building a full backend on?
#+end_quote

* The end game: the UI for valis

This backend is not an end in itself.  It is the rendering substrate for
the *valis user interface* - the "face" of valis as, in Fade's words, "an
actual computer over network protocols", not just another hosting play.
The ambition is a *Lisp-machine-style UI delivered over the web*: the thing
that distinguishes valis from a hosting product is that you operate it like
a computer, through a live, typed, object-oriented interface, over the
network.

Shape of the target system (Fade's framing, 2026-06):

- *Transport + rendering:* a *websocket <-> CLIM/WebGL* pipeline.  CLOG
  supplies the websocket transport and the drawing surface (2D canvas now,
  with ~clog-webgl~ available as a later target); ~clim-clog~ maps CLIM
  onto it.  Phases 0-2 here establish the 2D-canvas path; WebGL is a future
  rendering target for the same medium protocol.
- *Object-capability mediation:* every interaction is to be mediated by
  valis' *OCAP* (object-capability) interface.  Commands act on remote
  objects through capabilities, never ambient authority.  This dovetails
  with CLIM's presentation model (below): a presentation already ties a
  visual to a typed application object - under valis that reference is a
  capability.
- *Interaction flow via continuations:* UCW's ~call~/~answer~ continuation
  model (delimited continuations via a CPS code walker) is the leading
  candidate for multi-step flows, mirroring CLIM's ~accepting-values~ and
  command loop.  The heavy piece is ~arnesi~'s code walker, which can
  likely be pulled out as a standalone dependency rather than adopting all
  of UCW.  (See the UCW reference under Phase 2.)
- *Sibling repositories / naming:* valis' protocol plugins live in a repo
  Fade is calling *c3po* (a protocol droid).  This UI is a sibling effort
  in the same family; Glenn leads it ("czar of the UI").  The repository
  name is still to be chosen - see "Naming" below.

* CLIM in context (why CLIM, culturally)

CLIM (Common Lisp Interface Manager) is the standardised descendant of the
*Lisp Machine* user-interface toolkits - Symbolics Genera's Dynamic
Windows.  It was specified as CLIM I/II in the early 1990s (Symbolics,
Franz, International Lisp Associates).  McCLIM is the maintained
free-software implementation, and ~clim-clog~ is a McCLIM /backend/, so any
CLIM application - including valis' UI - can target the browser unchanged.

Its defining idea is the *presentation*: displayed output is semantically
typed, linking a visual representation to an application object /and its
presentation type/.  From this one idea fall out:

- *context-sensitive input* - click any displayed object whose type the
  current command wants;
- the *command loop* and command processor;
- ~accepting-values~ dialogs (the continuation-flavoured input flow);
- *incremental redisplay* over live objects.

Culturally this is the Lisp-machine worldview: the system is a space of
live, typed, inspectable objects, and the UI is /generated from data and
commands/ rather than hand-placed widgets.  That is precisely the idiom
valis wants - "an actual computer over network protocols" - which is why
Fade says "culturally, CLIM feels like the right thing", and why the goal
is a faithful CLIM backend rather than a bespoke web widget toolkit.

* Background: how a McCLIM backend is structured

A McCLIM backend implements a handful of protocol classes. We model ours on
the in-tree ~Backends/Null~ skeleton and the turtleware McCLIM-backends
tutorial series (https://turtleware.eu/tag/clim.html):

- *Port* - the logical display connection. CLX wraps an X11 socket; we wrap
  a CLOG connection + a ~<canvas>~ and its 2D context. Registered for the
  ~:clog~ server-path designator via ~find-port-type~.
- *Graft* - the "root window" sheet; reports the display dimensions. For us,
  the canvas size.
- *Medium* - holds the drawing context for a sheet; the ~medium-draw-*~
  generic functions are specialised here. *This is the core of the mapping.*
  ~medium-drawable~ returns the CLOG 2D context.
- *Frame manager* - realises application frames/panes. Minimal for now.

* The CLIM -> CLOG drawing map

The key insight that makes CLOG a strong fit: the HTML5 Canvas 2D model is
/imperative path drawing/, which is exactly CLIM's medium model. The
tutorial emits JS strings; we call CLOG methods instead.

| CLIM medium op         | CLOG canvas call(s)                                          |
|------------------------+-------------------------------------------------------------|
| ~medium-draw-line*~    | ~begin-path~ + ~move-to~ + ~line-to~ + ~path-stroke~        |
| ~medium-draw-lines*~   | per segment: as above                                       |
| ~medium-draw-polygon*~ | ~begin-path~ + ~move-to~/~line-to~ + (~close-path~) + fill/stroke |
| ~medium-draw-rectangle*~ | ~fill-rect~ / ~stroke-rect~                               |
| ~medium-draw-point(s)*~ | ~arc~ + ~path-fill~                                        |
| ~medium-draw-ellipse*~ | unit-circle ~arc~ under an affine ~transform~ (exact, incl. skewed) |
| ~medium-draw-text*~    | set ~font-style~ + ~fill-text~                              |
| ~medium-clear-area~    | ~clear-rect~                                                |
| ink (solid colour)     | ~(setf (stroke-style/fill-style ctx) "rgb(...)")~           |
| text style             | CSS ~font~ shorthand string                                 |
| text metrics           | ~measure-text~ round-trip, cached (offline estimate fallback) |

* Roadmap

** Phase 0 - output-only spike (DONE, v0.0.1)

Prove the medium mapping by drawing a static scene to the canvas. No
windowing, no input. Entry point: ~clim-clog::start~.

** Phase 1 - real top-level sheets & frames

- ~realize-mirror~ creates a canvas per mirrored (top-level) sheet.
- Frame manager lays out panes; map CLIM gadgets either to McCLIM's own
  pane-drawn gadgets (drawn through our medium) or to CLOG DOM widgets.
- Make an actual ~clim:application-frame~ display on a CLOG page.

** Phase 2 - input (DONE, the make-or-break for interactivity)

*Status: working.* Browser pointer events drive the CLIM event loop end to
end, confirmed visually (clicking the demo drops markers at the click).

- ~clog:set-on-mouse-down/up/move~ handlers, installed on each mirror's
  canvas in ~realize-mirror~, build CLIM events and push them onto a
  lock-protected queue on the port (see ~src/input.lisp~).
- ~initialize-instance~ calls ~restart-port~ to start McCLIM's port I/O
  loop (a thread looping ~process-next-event~) - the single queue consumer.
- ~process-next-event~ pops one event and ~distribute-event~s it into the
  sheet hierarchy, reaching ~handle-event~ on the target pane.
- Confirmed path: browser -> CLOG handler (worker thread) -> CLIM event ->
  port queue -> port I/O loop -> distribute-event -> handle-event -> medium
  draw -> browser.

The full presentation/command loop now works: clicking a /presented/ object
runs its command, and *hover highlighting* works too.  Two backend fixes were
needed beyond raw event delivery:
- ~clog-pointer~ (a ~standard-pointer~ subclass) caches the live cursor
  position, updated from every pointer event in ~make-clog-pointer-event~.
  McCLIM's ~standard-pointer~ hardcodes ~pointer-position~ to (0,0) and errors
  on setf - the backend is expected to track it.  Presentation highlighting /
  pointer-documentation /synthesize/ a motion event at ~pointer-position~, so
  without this the tracker always probed (0,0) and nothing highlighted.
- The medium now honours the device clipping region (~with-clipped-context~
  via ~medium-device-region~).  Partial repaints (e.g. un-highlighting a
  presentation) replay the output history clipped to a small region; without
  clipping a full-extent record (the demo's background rectangle) repainted
  the whole canvas and erased everything outside the region.

Remaining for a later pass: keyboard events (canvas needs focus/tabindex).

*Redisplay flashing - FIXED in Phase 3.* ~redisplay-frame-pane~ re-runs the
display function, and each CLOG drawing op renders individually over the
websocket, so a full redisplay used to visibly flash (whole canvas clears,
then content pops back in).  Native backends hide this behind windowing-system
buffering; a canvas has none.  This is now solved by *double buffering* (draw
to an offscreen canvas, blit to the visible one in a single op per flush) -
see Phase 3 below.

*Reference - UCW continuations.* [[https://github.com/erikg/ucw-core][UCW (UnCommon Web)]], in Quicklisp and still
running on modern SBCL, models multi-page web flows as straight-line code
using delimited continuations (~arnesi~'s CPS walker): a component can
~call~ another and later ~answer~ a value back, suspending/resuming across
HTTP requests. This is the same shape as CLIM's ~accepting-values~ dialogs
and the command loop ("ask the user, suspend, resume with a value"). If the
naive event-injection approach to the command loop proves awkward, UCW's
~call~/~answer~ is the model to borrow for input flow -- complementary to
CLOG, which handles transport + drawing. (Control flow, not rendering: UCW
is not a drawing surface and does not replace the CLOG medium.)

** Phase 3 - drawing fidelity

- *Double buffering (DONE).* Each mirror now owns a hidden offscreen canvas
  (~clog-mirror-offscreen-canvas~ / ~-context~).  ~medium-drawable~ returns
  the offscreen context, so all CLIM drawing accumulates off-screen;
  ~medium-finish-output~ / ~medium-force-output~ blit it to the visible
  canvas with a single ~draw-image~ (~clog-mirror-flush~).  The visible
  context is set to the ~"copy"~ composite mode once at ~realize-mirror~ (and
  re-applied after any ~set-mirror-geometry~ resize, which resets context
  state), so the blit /replaces/ the visible canvas exactly -- including
  cleared/transparent regions from ~window-clear~ -- in one op, with no
  intermediate clear that would flash.
- *Real text metrics (DONE).* ~text-size~, ~text-style-ascent/descent/height~,
  ~text-style-width~ and ~text-style-character-width~ now round-trip
  ~clog:measure-text~ over the websocket: font-level ascent/descent come from
  ~font-bounding-box-ascent/descent~, widths from the measured advance.
  Results are cached on the port (~clog-port-text-metrics-cache~), keyed by
  the CSS-font string (font metrics) and by ~(css . char)~ (per-glyph width),
  since browser metrics are stable for the life of a connection.  When no
  drawable context exists yet (space composition before the canvas is
  realized) the old offline estimate is used as a fallback and not cached.
  *Crucially, ~text-size~ computes a string's width as the sum of cached
  per-glyph widths (~%line-width~) rather than measuring the whole substring*:
  ~text-size~ is called on the growing input line on /every/ keystroke during
  DREI redisplay, so measuring the substring blocked on a websocket round-trip
  per keystroke (the cause of interactor input lag).  Summing cached glyph
  widths costs one ~measure-text~ per distinct character ever, then zero.
- *Text drawing transformation (gotcha, fixed).* Unlike every other
  ~medium-draw-*~ generic, McCLIM's ~transform-coordinates-mixin~
  ~:around~ method for ~medium-draw-text*~ is disabled (guarded by ~#+(or)~ in
  =Core/drawing/medium.lisp=), so X/Y arrive in the medium's /user/ coordinate
  system with the medium transformation NOT applied.  DREI / output-record
  replay positions a text run by setting the medium transformation to
  translate to the run origin and drawing the string at (0,0); our
  ~medium-draw-text*~ must therefore apply ~medium-transformation~ itself
  before the sheet native transformation.  Without this, every text run landed
  at the pane origin -- outside its own clip rectangle -- so e.g. typed
  interactor text was invisible while the caret (a shape, which /does/ get the
  ~:around~) moved correctly.
- *True multi-rectangle clipping (DONE).* ~with-clipped-context~ now builds one
  canvas path from /every/ rectangle of the device region
  (~map-over-region-set-regions~) before ~path-clip~, so a ~standard-rectangle-set~
  clips to the union of its members rather than to a single bounding rectangle.
  Non-rectangular sub-regions still fall back to their bounding box.
- *Line dashes (DONE).* ~apply-stroke-state~ pushes ~line-style-dashes~ to the
  canvas via ~set-line-dash~ (NIL = solid, T = default pattern, or a sequence of
  segment lengths), reset on every stroke.
- *General (skewed) ellipses (DONE).* Drawn by installing the affine map that
  sends the unit circle to the ellipse (~transform~ with the two radius vectors
  as columns) and tracing a unit ~arc~; exact for non-orthogonal radii.
- *Pixmaps + ~medium-copy-area~ (DONE).* A ~clog-pixmap~ is a hidden off-screen
  ~<canvas>~ + 2D context; ~allocate-pixmap~/~deallocate-pixmap~ manage it and
  ~medium-drawable~ honours a pixmap bound via ~with-output-to-pixmap~.  All four
  ~medium-copy-area~ combinations (medium/pixmap x medium/pixmap) move pixels
  with ~get-image-data~ + ~put-image-data~ (CLOG's ~draw-image~ has no source
  sub-rectangle form).  This backs ~copy-area~, ~copy-to/from-pixmap~ and
  ~with-output-to-pixmap~.
- Gradients/patterns as CLIM design inks.  ~(setf fill-style)~ accepts a CLOG
  gradient/pattern object, but ~(setf stroke-style)~ does not, and CLIM's
  indirect/tiled inks need rendering to an off-screen pattern; deferred as it
  is rarely exercised by basic applications.  *(still to do)*

** Interactive performance (best practice)

The transport is a websocket, so the cost model is not "how many draw ops"
but "how many /synchronous round-trips/":

- *Drawing ops are fire-and-forget.* CLOG sends ~begin-path~, ~fill-rect~,
  ~fill-text~, the double-buffer ~draw-image~ flush, etc. without waiting for
  a reply, so they pipeline cheaply.  Redrawing a whole pane of shapes per
  event is fine.
- *Some ops block on a reply.* ~get-image-data~ (used by ~medium-copy-area~,
  ~copy-area~, and pixmap blits) and ~measure-text~ (~text-size~) do a full
  request/response round-trip.  Issuing one of these /per keystroke or per
  pointer-motion/ is what makes interaction feel sluggish.
- *Mitigations:* (1) keep ~measure-text~ cached per font/glyph AND build
  string widths by summing cached glyph widths, never measuring a whole
  substring on the hot path (done -- see ~clog-port-text-metrics-cache~ and
  ~%line-width~/~text-size~); (2) in interactive handlers, repaint only
  the region that changed rather than re-running a full display function that
  contains a ~copy-area~/pixmap step; (3) reserve ~copy-area~ for genuine
  scrolling, not per-frame effects.  The showcase demo (~start-showcase~)
  illustrates this: its key/click handlers repaint a single row, while the
  full scene (which includes a pixmap + ~copy-area~) is drawn only on
  ~handle-repaint~.

For a presentation-dense UI (ubik), pointer-motion highlighting is the case
to watch -- but it is *already responsive* (verified): a ~mousemove~ enqueues
a ~pointer-motion-event~ (browser -> server, one-way), CLIM hit-tests it, and
the highlight is drawn with the same fire-and-forget ops as everything else,
so there is *no blocking round-trip per hover*.  The Phase-2 enabler is the
~clog-pointer~ that tracks the live cursor position (~standard-pointer~
hardcodes (0,0), which would make the tracker always probe the origin and
highlight nothing).  Client-side hit-testing (see Open questions) remains an
*optional* future micro-optimization -- to avoid even sending motion events /
doing server-side hit-testing -- not a correctness or responsiveness blocker.

** Phase 4 - the listener / editor

- Reuse CLOG's listener + editor as the "Lisp machine" shell around CLIM

* Open questions / risks

- *Input latency (RESOLVED / verified).* Pointer-motion-driven presentation
  highlighting over the websocket is responsive: motion events are one-way
  (browser -> server) and highlights draw with fire-and-forget ops, so there
  is no blocking round-trip per hover (verified with ~start-present-demo~).
  Client-side hit-testing remains an optional future micro-optimization, not a
  blocker.
- *Synchronous text metrics (RESOLVED).* CLIM wants ~text-size~ synchronously
  during layout; the browser has the real answer.  We measure per font/glyph
  via ~clog:measure-text~ once and cache it, and build string widths by
  summing cached glyph widths (~%line-width~) -- so the per-keystroke hot path
  does no synchronous round-trips at all.
- *Per-connection state.* Each CLOG connection is a stateful session; a CLIM
  frame is naturally single-session, which fits, but multi-user is out of
  scope for the spike.

* Comparison note (CLOG vs Fluxion)

For a /faithful/ CLIM backend (this project), CLOG is the natural substrate:
bidirectional websockets match CLIM's event loop and ~clog-canvas~ is a
direct medium target. Fluxion's FRP + SSE + DOM-morphing model is excellent
for declarative web apps but is a paradigm mismatch for CLIM's imperative,
pointer-driven, retained-graphics model - it would mostly be reduced to a
transport. Fluxion would shine for a /different/ goal: a new, CLIM-inspired
/native-HTML/ toolkit (presentations -> components, value gadgets ->
reactive cells), which does not run existing McCLIM apps.
