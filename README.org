#+TITLE: clim-clog
#+OPTIONS: toc:nil num:nil

A [[https://mcclim.common-lisp.dev/][McCLIM]] backend that renders to an *HTML5 ~<canvas>~ in a web browser*,
driven over a [[https://github.com/rabbibotton/clog][CLOG]] websocket connection.

The goal is to investigate putting CLIM applications on the web by reusing
CLOG as the transport + drawing surface, instead of inventing a new one.
This is the concrete spike that came out of the "map CLIM onto websockets +
CLOG" discussion.

* Status

*Phases 0-2 done, Phase 3 in progress: a CLIM application frame runs
interactively in the browser, with the presentation/command loop working,
double-buffered (no redisplay flash) and using real browser text metrics.*

| Area                                                                            | State                               |
|---------------------------------------------------------------------------------+-------------------------------------|
| Port / graft / frame-manager skeleton                                           | done (modelled on ~Backends/Null~)  |
| Medium drawing (line, lines, polygon, rectangle(s), point(s), ellipse, text)    | done, delegating to CLOG canvas     |
| Solid-colour inks                                                               | done                                |
| Coordinate transformations                                                      | applied per-op                      |
| Mirrored top-level sheets / real frames & panes (Phase 1)                       | done                                |
| Input - pointer & keyboard (Phase 2)                                            | done                                |
| Presentation / command loop + hover highlighting                                | done                                |
| Device-region clipping (bounding-rectangle)                                     | done                                |
| Double buffering (no redisplay flash) (Phase 3)                                 | done                                |
| Real browser text metrics (cached ~measure-text~) (Phase 3)                     | done                                |
| True multi-rectangle / non-rectangular clipping                                 | *not yet* (Phase 3)                 |
| Line dashes, skewed ellipses, gradients/patterns, pixmaps                       | *not yet* (Phase 3)                 |

See [[file:DESIGN.org][DESIGN.org]] for the architecture and full roadmap.

* Why CLOG

~clog-canvas.lisp~ is a near 1:1 Lisp wrapper over the HTML5 Canvas 2D API
(~begin-path~, ~move-to~, ~line-to~, ~arc~, ~path-fill~, ~path-stroke~,
~fill-text~, ~measure-text~, transforms, ...). Where the turtleware
McCLIM-backends tutorial emits JavaScript /strings/, here we call the equivalent
CLOG method and let CLOG marshal it over the websocket. CLOG's
bidirectional connection also gives us the pointer/keyboard events we will
need for interactivity in Phase 2, and it already ships a listener/editor
(CLOG Builder).

* Dependencies

- A Common Lisp with McCLIM available (~clim-core~).
- CLOG (~clog~) - via Ultralisp or git.

Local clones used during development live at ~~/SourceCode/McCLIM~ and
~~/SourceCode/clog~ (kept current with their upstreams).

* Running the demos

Each entry point starts the CLOG server and opens a browser tab.  Call
~(clim-clog::stop)~ to shut the server down between demos.

#+begin_src lisp
(ql:quickload :clim-clog/examples)

;; Phase 0 - bare medium: a static scene drawn straight through MEDIUM-DRAW-*.
(clim-clog::start)              ; rectangles, line, polygon, ellipse, points, text

;; Phase 1 - a real CLIM application frame on a CLOG page.
(clim-clog::start-frame)

;; Phase 2 - input.
(clim-clog::start-input-demo)   ; click to drop a marker
(clim-clog::start-key-demo)     ; click to focus, then type
(clim-clog::start-present-demo) ; click a presented shape to run its command

(clim-clog::stop)
#+end_src

The spike's original question is answered: *yes, a CLIM medium can drive a
browser canvas through CLOG* - and, as of Phase 2, the full presentation /
command loop runs interactively over the websocket.
